## 0. Цель обзора

Показать, что **ни один существующий инструмент не сопоставляет нормативную методику измерения (ГОСТ Р 8.563-2014 и аналоги) с фактической трассой исполнения программного сценария ИИС**.

Анализируемое поле:
- Категория А — индустриальные инструменты экосистемы National Instruments (ближайший контекст разработки ИИС).
- Категория Б — академические/open-source библиотеки runtime verification на базе STL/MTL/MFOTL.
- Категория В — формальные методы верификации (timed automata) как теоретический фундамент.

Каждый инструмент описан по 14 полям (см. ниже). Поля, по которым достоверной информации не нашлось в открытом доступе на дату обзора, помечены `[NOT FOUND]`.

## 2. Категория Б — Runtime Verification и Temporal Logic

В этой категории — академические и open-source инструменты, которые **технологически близки к FLV**: умеют сопоставлять трассу/сигнал с формальной спецификацией. Главное отличие — у них нет прикладной DSL под измерительную методику ИИС.

## 5. Выводы (для главы 1.3 ПЗ)

**5.1 Что закрывают существующие инструменты.**
Индустрия NI закрывает три задачи: (а) статический анализ кода test-секвенций и LabVIEW VI на стиль и типовые ошибки (TestStand Sequence Analyzer, VI Analyzer); (б) трассируемость требований к артефактам разработки на уровне идентификаторов и регулярных выражений (Requirements Gateway); (в) модульное функциональное тестирование отдельных VI (Unit Test Framework). Академическое поле runtime verification предоставляет мощные формальные языки — STL, MTL, MFOTL — и инструменты их мониторинга на сигналах/event-log'ах (RTAMT, Breach, py-metric-temporal-logic, MonPoly), а также инструменты обучения STL-формул из примеров (TeLEx). Формальный фундамент тайм-автоматов закрыт UPPAAL.

**5.2 Что эти инструменты НЕ закрывают (наша ниша).**
Ни один из проанализированных инструментов **не работает на уровне нормативной измерительной методики как отдельной сущности**. У NI инструментов нет понятия «шаг методики», «гард стабилизации», «обязательность точки» — они анализируют код, не семантику измерения. У академических STL/MTL/MFOTL инструментов есть мощные общие формализмы, но **отсутствует прикладной DSL**, описывающий методику измерения в терминах, совместимых с ГОСТ Р 8.563-2014 и ГОСТ 8.207-76. Никто не предоставляет **end-to-end путь**: «нормативный документ → формальная модель методики → автоматическое сопоставление с фактической трассой исполнения сценария ИИС → структурированный verdict с привязкой нарушения к пункту методики».

**5.3 Прямые отличия FLV от ближайших аналогов.**
1. **Уровень абстракции**: FLV работает с понятиями измерительной методики (шаг, гард, предикат стабилизации, обязательность точки), а не с сигналами или общими STL-формулами.
2. **Прикладной DSL**: FLV вводит DSL, привязанный к нормативке (ГОСТ Р 8.563-2014), а не к общим темпоральным логикам. Это снижает порог входа для метролога без CS-образования.
3. **Сопоставление спецификации ↔ трассы**: FLV закрывает зазор между «декларативной методикой» и «event-log'ом исполнения», который ни один NI-инструмент и ни один STL-монитор по отдельности не закрывают.
4. **Структурированный каталог нарушений**: FLV выдаёт verdict с типами нарушений (SEQ_MISS, SEQ_ORDER, GUARD_FAIL, TIME_VIOL и т.д.), привязанными к шагам методики и пунктам ГОСТ — этого нет ни у RTAMT (бинарный/количественный verdict по STL), ни у NI Sequence Analyzer (правила про code style), ни у Requirements Gateway (только coverage-метрика).
5. **Применимость в ИИС**: FLV проектируется как процессный инструмент в составе ИИС (или рядом с ним), а не как code review tool (NI) или академический monitor общего назначения. Целевой класс — программные модели измерительных процессов в ИИС, что нигде явно не покрыто.

**5.4 Семантический разрыв (тезис для главы 1).**
Тезис: между **нормативной измерительной методикой** (текстовый ГОСТ + методические указания) и **программной реализацией измерительного процесса в ИИС** существует семантический разрыв, который текущие инструменты не закрывают. NI-инструменты сосредоточены на качестве кода, академические инструменты — на формальных свойствах сигналов; ни одна из этих линий не предоставляет специализированного механизма, который бы (i) формализовал методику в виде машиночитаемой DSL-спецификации, (ii) исполнял программный сценарий ИИС с записью структурированной трассы, (iii) автоматически проверял соответствие трассы спецификации с привязкой нарушений к пунктам нормативного документа. Закрытие этого разрыва и есть содержательная новизна метода FLV.
