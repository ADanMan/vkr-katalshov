# Формат event-log (JSONL)

> Phase 2. Дата: 2026-05-03. Автор: Катальшов Д.А., К2-81Б. Документ описывает формат журнала событий, который пишется логгером ИИС в ходе исполнения программного измерительного сценария и подаётся на вход FLV-модуля. Сопровождается JSON Schema (`event_log.schema.json`).

---

## 1. Назначение

Event-log — машинно-читаемый журнал, фиксирующий ход исполнения сценария измерения с гранулярностью, достаточной для функционально-логической верификации. Источник лога — hook-логгер, встроенный в контроллер сценария ИИС (см. `02_Спецификация/positioning.md`, точка [3a]).

Формат — **JSON Lines (JSONL)**: одна строка = один JSON-объект = одно событие. Преимущества JSONL:
- потоковая запись без необходимости держать весь файл в памяти;
- удобный line-oriented анализ Unix-инструментами;
- расширяемость (новые поля не ломают парсер);
- совместимость с готовыми библиотеками (pandas `read_json(lines=True)`, jq).

Расширение файла: `.jsonl`. Кодировка: UTF-8.

---

## 2. Общая структура события

Каждая строка лога — JSON-объект следующего формата:

```json
{
  "ts": "2026-05-03T14:23:45.123Z",
  "stand_id": "S1",
  "run_id": "S1-20260503-001",
  "seq": 42,
  "state": "HOLD",
  "event": "HOLD_END",
  "params": {"t_hold": 285.4, "dT_dt": 0.035},
  "signals": {"T": 150.7},
  "errors": []
}
```

### Обязательные поля

| Поле | Тип | Описание |
|---|---|---|
| `ts` | string (ISO-8601) | Метка времени UTC с миллисекундным разрешением. Формат: `YYYY-MM-DDTHH:MM:SS.sssZ`. |
| `stand_id` | string | Идентификатор стенда (`S1`, `S2` и т.д.). |
| `run_id` | string | Идентификатор прогона: `<stand>-<YYYYMMDD>-<NNN>`. Уникален для прогона. |
| `seq` | integer ≥ 0 | Монотонно возрастающий порядковый номер события в пределах прогона (стартует с 0). |
| `state` | string | Текущее состояние FSM (`INIT`, `HEAT`, `HOLD`, `MEASURE`, `POST` для S1). Значения совпадают с `states[].name` в DSL. |
| `event` | string | Имя события (см. таблицу ниже). |

### Опциональные поля

| Поле | Тип | Описание |
|---|---|---|
| `params` | object | Параметры процесса в момент события (`t_hold`, `T_set`, `N`, и пр.). |
| `signals` | object | Снимок измерительных сигналов в момент события (`T`, `dT_dt`, `R`, и пр.). |
| `errors` | array of object | Список ошибок прибора/драйвера, если есть. |
| `meta` | object | Произвольные метаданные (версия сценария, оператор, заметки). |

## 3. Каталог событий

| Имя события | Когда срабатывает |
|---|---|
| `RUN_START` | Старт прогона (первое событие). |
| `INIT_START` / `INIT_END` | Вход / выход из состояния INIT. |
| `HEAT_START` / `HEAT_END` | Вход / выход из состояния HEAT. |
| `HOLD_START` / `HOLD_END` | Вход / выход из состояния HOLD. |
| `MEAS_START` / `MEAS_END` | Вход / выход из состояния MEASURE. |
| `MEAS_TICK` | Один отсчёт в режиме MEASURE (одно из N = N_min измерений). |
| `POST_START` / `POST_END` | Вход / выход из состояния POST. |
| `STATE_TRANS` | Универсальное событие перехода: содержит `from`, `to`, `transition_id` в `params`. Альтернатива `<STATE>_END` + `<STATE>_START`. |
| `SAMPLE` | Внутрисостоянный отсчёт (например, температура каждую секунду). Содержит `signals`. |
| `SET_RANGE_OK` | (для SCPI-стендов) подтверждение установки диапазона. |
| `SETTLE_START` / `SETTLE_END` | (для SCPI-стендов) settling-окно. |
| `OPC_QUERY` | (для SCPI-стендов) `*OPC?` — синхронизация. |
| `ERROR` | Ошибка прибора, драйвера или логгера. Содержит `errors[]`. |
| `WARN` | Нефатальное предупреждение. |
| `RUN_END` | Завершение прогона (последнее событие). Содержит `params.run_status` ∈ {`completed`, `aborted`, `error`}. |

Для S1 минимально достаточный набор событий: `RUN_START`, `INIT_START/END`, `HEAT_START/END`, `HOLD_START/END`, `MEAS_START`, `MEAS_TICK` (N раз), `MEAS_END`, `POST_START/END`, `RUN_END`. Опциональное `SAMPLE` каждую секунду в HOLD/HEAT для диагностики.

### 4.1 Корректный прогон S1 (фрагмент)

```jsonl
{"ts":"2026-05-03T10:00:00.000Z","stand_id":"S1","run_id":"S1-20260503-001","seq":0,"state":"INIT","event":"RUN_START","params":{"T_set":150,"t_hold_min":300,"N_min":20},"signals":{"T":24.3}}
{"ts":"2026-05-03T10:00:00.500Z","stand_id":"S1","run_id":"S1-20260503-001","seq":1,"state":"INIT","event":"INIT_END","params":{"device_ready":true},"signals":{"T":24.5}}
{"ts":"2026-05-03T10:00:00.500Z","stand_id":"S1","run_id":"S1-20260503-001","seq":2,"state":"HEAT","event":"HEAT_START","params":{},"signals":{"T":24.5}}
{"ts":"2026-05-03T10:02:34.700Z","stand_id":"S1","run_id":"S1-20260503-001","seq":312,"state":"HEAT","event":"HEAT_END","params":{"T_reached":50.1},"signals":{"T":50.1}}
{"ts":"2026-05-03T10:02:34.700Z","stand_id":"S1","run_id":"S1-20260503-001","seq":313,"state":"HOLD","event":"HOLD_START","params":{},"signals":{"T":50.1}}
{"ts":"2026-05-03T10:08:14.700Z","stand_id":"S1","run_id":"S1-20260503-001","seq":653,"state":"HOLD","event":"HOLD_END","params":{"t_hold":340,"dT_dt":0.018},"signals":{"T":150.2}}
{"ts":"2026-05-03T10:08:14.700Z","stand_id":"S1","run_id":"S1-20260503-001","seq":654,"state":"MEASURE","event":"MEAS_START","params":{},"signals":{"T":150.2}}
```

### 4.2 Прогон с инъекцией TIME_UNDER (фрагмент)

```jsonl
{"ts":"2026-05-03T11:00:34.700Z","stand_id":"S1","run_id":"S1-20260503-002","seq":313,"state":"HOLD","event":"HOLD_START","params":{},"signals":{"T":50.1}}
{"ts":"2026-05-03T11:01:04.700Z","stand_id":"S1","run_id":"S1-20260503-002","seq":343,"state":"HOLD","event":"HOLD_END","params":{"t_hold":30,"dT_dt":0.045},"signals":{"T":80.5}}
```

Здесь `t_hold = 30` (вместо ≥ 300) и `dT_dt = 0.045` (вместо ≤ 0.02) — FLV-matcher должен поднять `TIME_UNDER` и `PRED_FAIL`.

## 5. Принципы реализации логгера в ИИС

1. **Append-only.** Никогда не перезаписываем уже записанную строку.
2. **Atomic line write.** Каждая строка пишется атомарно (одним системным вызовом write), чтобы при аварийном завершении лог был валидным до последней полной строки.
3. **fsync на критичных событиях.** `RUN_END`, `ERROR` — с принудительной синхронизацией на диск.
4. **Async-friendly.** В LabVIEW — через Producer-Consumer / Queue Operations; в Python — через `asyncio` или отдельный поток.
5. **Sampling.** Внутрисостоянные `SAMPLE`-события могут сэмплироваться, чтобы не перегрузить лог. По умолчанию — 1 раз/с в HOLD/HEAT.
6. **Versioning.** В первом `RUN_START` — поле `meta.log_format_version` = `"1.0"`.

---

## 6. Совместимость с DSL

Имена `state` в логе **должны** совпадать с `states[].name` в DSL. Имена `event` — из каталога выше; новые события добавляются обновлением документа + Schema.

Параметры в `params` могут включать любые ключи; FLV-matcher использует только те, которые упомянуты в `transitions[].guard`, `checks[].condition`, `parameters` в DSL.

## 7. Связь с компонентами FLV

| Компонент FLV | Что делает с event-log |
|---|---|
| `log_parser.py` | Читает JSONL построчно, валидирует по Schema, нормализует временные метки в относительные секунды от RUN_START. |
| Sequence matcher | По event'ам STATE_TRANS / `<STATE>_START` / `<STATE>_END` восстанавливает фактическую последовательность переходов и сверяет с FSM в DSL. |
| Timing matcher | По временам START/END каждого состояния считает фактическую длительность; сверяет с `transitions[].time` и `checks[kind=timing]`. |
| Predicate matcher | По `params` и `signals` в момент перехода вычисляет предикаты из `transitions[].guard` и `checks[kind=predicate]`. |
| Reporter | Включает в отчёт фрагменты event-log как контекст для обнаруженных нарушений. |

## 8. Ограничения версии 1.0

- Поддержка только дискретных событий + редких SAMPLE. Полноценный непрерывный сигнал (для STL-анализа) — в версии 1.1 (post-defense).
- Один поток событий на прогон. Multi-stand параллелизм — в версии 1.1.
- Без шифрования и подписи. Для production-ИИС — добавить hash-chain в meta для tamper-detection (post-defense).
