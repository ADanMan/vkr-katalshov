# Позиционирование FLV в архитектуре ИИС

> Phase 2. Дата: 2026-05-03. Документ объясняет: что именно проектируется, в какой блок жизненного цикла разработки и эксплуатации ИИС встраивается метод FLV, какой эффект ожидается. Прямой вход для введения и главы 1-2 ПЗ.

---

## 1. Что мы проектируем

**FLV (Functional-Logical Verification) — метод и программный инструмент** для автоматической проверки соответствия фактической трассы исполнения программного измерительного сценария формализованной нормативной модели методики.

Технически: **гибридная LLM-формальная система**. LLM применяется на шаге извлечения нормативной модели из неструктурированного текста; детерминированный конечно-автоматный matcher — на шаге сопоставления трассы с моделью; LLM применяется повторно на шаге формирования объяснения вердикта (см. `99_Артефакты/ADR_002_role_of_LLM.md`).

**Не проектируем:**
- Не само измерение (не сенсоры, не АЦП, не математическая обработка результатов).
- Не новый язык программирования сценариев.
- Не полнофункциональный замениватель TestStand/LabVIEW.
- Не «универсальный AI-ассистент» — LLM работает в **строго ограниченных** ролях с детерминированной перепроверкой.

## 2. Типовая блок-структура ИИС и место FLV

```
┌─────────────────────────────────────────────────────────────────┐
│                     ЖИЗНЕННЫЙ ЦИКЛ ИИС                          │
└─────────────────────────────────────────────────────────────────┘

  [1] МЕТОДИКА ИЗМЕРЕНИЙ (текстовый документ ГОСТ/МИ/ТУ)
              │
              │  «семантический разрыв» — наша проблемная зона
              ▼
  [2] ПРОГРАММНАЯ МОДЕЛЬ СЦЕНАРИЯ (LabVIEW VI / TestStand seq / Python)
              │
              ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  [3] ИСПОЛНЕНИЕ В ИИС:                                      │
  │      [3a] контроллер сценария (state machine / QSM)         │
  │      [3b] драйверы приборов (SCPI / VISA / DAQmx)           │
  │      [3c] первичные преобразователи (PT100 / АЦП и т.п.)    │
  │      [3d] логгер событий и измерений                        │
  └─────────────────────────────────────────────────────────────┘
              │
              ▼
  [4] АНАЛИЗ И ОТЧЁТНОСТЬ
              │
              ▼
  [5] МЕТРОЛОГИЧЕСКАЯ ПРОСЛЕЖИВАЕМОСТЬ
```

**FLV встраивается между блоками [1]→[2] и [3]→[4]:**

```
[1] МЕТОДИКА (текст) ───── [LLM: извлечение в DSL] ──── [DSL spec] (формальная нормативная модель)
                                                              │
                                                              │
                                                              ▼
[3a] контроллер сценария ──── [hook: event-logger] ──── [JSONL trace]
                                                              │
                                                              ▼
                                          ┌──────── [FLV matcher] ─────────┐
                                          │      sequence + timing +       │
                                          │      predicate проверки        │
                                          │      детерминированно          │
                                          └────────────┬───────────────────┘
                                                       │
                                                       ▼
                                          [verdict: OK / FAIL + violations]
                                                       │
                                                       ▼
                          [LLM: human-readable объяснение, привязка к пунктам ГОСТ]
                                                       │
                                                       ▼
                                          [4] АНАЛИЗ / [5] прослеживаемость
```

**Ключевые точки интеграции в ИИС:**

| Точка | Что делает FLV | Когда срабатывает |
|---|---|---|
| `[1]→spec` | LLM-препроцессор: текст методики → DSL-скелет; инженер утверждает | один раз при подготовке методики |
| `[3a]→hook` | Логгер событий состояний и переходов FSM | в каждом прогоне |
| `[3]→matcher` | Сопоставление trace ↔ DSL → вердикт | offline (по окончании) ИЛИ online (потоково) |
| `verdict→[4]` | Привязка вердикта к ГОСТ/МИ, объяснение через LLM | один раз на прогон |
| `[5]` | Документ-вердикт хранится вместе с измерительным результатом | автоматически |

## 3. Где FLV вставляется относительно конкурентных инструментов

| Слой ИИС | Что покрывает существующий инструментарий | Где FLV |
|---|---|---|
| Стиль кода / читаемость VI | NI VI Analyzer Toolkit (статика) | вне зоны |
| Стиль test-секвенций | NI TestStand Sequence Analyzer (статика) | вне зоны |
| Трассируемость требования ↔ артефакт | NI Requirements Gateway | вне зоны (но FLV даёт **автоматизированную** трассировку реализации, а не ручную) |
| Юнит-тестирование VI | NI LabVIEW Unit Test Framework | вне зоны (FLV — интеграционный уровень) |
| Online STL-мониторинг сигналов | RTAMT, Breach | смежная зона; **FLV отличается DSL'ем для нормативной методики, а не сигналами** |
| Model checking тайм-автоматов | UPPAAL | вне зоны (FLV — runtime, а не дизайн-тайм) |
| LLM-валидация документов / кода | GitHub Copilot, ChatGPT-codereview | смежная зона; **FLV отличается формальным подкреплением вердикта детерминированным matcher'ом** |

**Уникальная позиция FLV:** связка «нормативная методика измерения → формальный DSL → реальная трасса исполнения сценария ИИС → объяснимый вердикт». Это не делает ни один из перечисленных инструментов в одиночку.

## 4. Эффекты (для введения и заключения ПЗ)

### Метрологические

- **Повышение метрологической прослеживаемости.** Каждый зафиксированный измерительный результат сопровождается формальным вердиктом FLV: соответствует ли исполнение нормативной методике. По требованиям ГОСТ Р 8.563-2009 это поддерживает аудит и сертификацию.
- **Снижение доли недостоверных результатов.** Прогоны, в которых FLV обнаружил нарушения (TIME_UNDER, PRED_FAIL и т.п.), — отбрасываются или повторяются автоматически. Целевое снижение — TPR детекции ≥ 0.9 при FPR ≤ 0.1 (см. метрики Phase 5).

### Инженерные

- **Раннее обнаружение семантических ошибок** в коде сценария: расхождения с методикой выявляются при первом же прогоне на стенде, а не после серии «правильно выглядящих, но неверных» измерений.
- **Снижение времени отладки методики.** Гипотеза, проверяемая в Phase 5: отладка измерительного сценария с FLV — на 30-50% быстрее, чем с ad-hoc проверками в коде.
- **Документированная привязка кода к нормативке.** LLM-объяснение вердикта ссылается на конкретные пункты ГОСТ/МИ.

### Организационные

- **Снижение зависимости от опыта оператора.** Junior-инженер, опираясь на FLV-вердикт, может верифицировать методику без глубокого знания LabVIEW/TestStand.
- **Воспроизводимость испытаний.** Один и тот же DSL — одинаковая верификация на разных стендах, в разных лабораториях, в разное время.

### Научные (новизна работы)

1. Формальная DSL-модель измерительных протоколов с явным каталогом нарушений, привязанных к нормативно-методическим требованиям.
2. Гибридная LLM-формальная архитектура: LLM на этапе извлечения и объяснения, детерминированный matcher на этапе вердикта. Каждая роль обоснована — LLM где силён, формальные методы где силны.
3. Метрики функционально-логической прослеживаемости (TPR, K_seq, J_timing) для количественной оценки соответствия программной реализации нормативной методике — новый инструментарий для метролога.

## 5. Что НЕ обещаем (threats to validity, для главы 4 ПЗ)

- FLV не верифицирует **физическую корректность** измерения (это задача поверки прибора, а не FLV).
- FLV не заменяет нормативную методику — только проверяет соответствие.
- LLM-извлечение DSL требует human-in-the-loop утверждения; полностью автоматическая работа невозможна без потери точности.
- FLV-метод привязан к адекватности модели в DSL: если DSL неполн, FLV пропустит соответствующее нарушение.

---

## 6. Связь с темой ВКР

Тема ВКР звучит: «Метод функционально-логической верификации программных моделей измерительных процессов в составе информационно-измерительных систем». Каждое ключевое слово темы покрыто:

- **«Метод»** — алгоритмическая основа FLV (DSL + Timed FSM + matcher pipeline + LLM-роли).
- **«Функционально-логической»** — функциональная (последовательность, условия) и логическая (предикаты, временные окна) проверка.
- **«Верификации»** — формальное сопоставление с нормативной моделью (а не валидация результата).
- **«Программных моделей»** — программные сценарии (LabVIEW/TestStand/Python) как объект проверки.
- **«Измерительных процессов»** — методика измерения как нормативный референс.
- **«В составе ИИС»** — встраиваемый в существующую ИИС, не отдельный заменяющий инструмент.

## 7. Минимальный сценарий применения (E2E demo для защиты)

1. Инженер вводит в систему текстовую методику (фрагмент ГОСТ Р 8.563): «Образец выдержать при T=150 °C не менее 300 секунд после стабилизации, после чего снять не менее 20 отсчётов с интервалом 1 секунда».
2. LLM парсит текст → генерирует DSL-скелет (`s1_stl_v1.yaml`) — состояния (HEAT/HOLD/MEASURE/POST), переходы, гарды (`T >= 150`, `t_hold >= 300`, `N >= 20`), параметры.
3. Инженер просматривает DSL, утверждает или редактирует.
4. Система генерирует/привязывается к LabVIEW VI (или Python-сценарию). VI запускается на стенде, логгер пишет JSONL-трассу.
5. FLV-matcher сопоставляет трассу с DSL → выдаёт вердикт.
6. Если FAIL — LLM формирует human-readable отчёт со ссылкой на пункт нормативного документа.
7. Корректные прогоны идут в архив с прикреплённым FLV-сертификатом соответствия.

Этот workflow — основа главы 2-3 ПЗ + презентационного demo Phase 7.
