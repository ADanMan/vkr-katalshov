# Формальная модель функционально-логической верификации

> Phase 2. Дата: 2026-05-03. Автор: Катальшов Д.А., К2-81Б. Документ задаёт математическую основу метода FLV: расширенный конечный автомат с временными ограничениями (Timed FSM) и предикатами, семантика «приёма» трассы, правила формирования вердикта. Прямой вход в главу 2 ПЗ.

---

## 1. Обозначения

В тексте используются следующие соглашения:
- Множества — заглавные буквы (𝒮, 𝒱, 𝒫, 𝒯).
- Элементы множеств — строчные (s, v, p, t).
- Кванторы — стандартные (∀, ∃).
- Логические связки — ∧, ∨, ¬, ⇒.
- ℝ⁺ — положительные действительные числа (включая 0).
- ℕ — натуральные числа (включая 0).

---

## 2. Структура нормативной модели

Нормативная модель измерительной методики формализуется как кортеж

> **𝓜 = ⟨𝒮, s₀, 𝒱, 𝒫, 𝒯, 𝒞, 𝒢, 𝒥⟩**

где:

| Символ | Что означает |
|---|---|
| **𝒮** | конечное множество **состояний** процесса (в S1: {INIT, HEAT, HOLD, MEASURE, POST}) |
| **s₀ ∈ 𝒮** | начальное состояние (для S1: INIT) |
| **𝒱** | конечное множество **переменных** процесса (в S1: T, dT_dt, t_hold, N, T_set, …) |
| **𝒫** | конечное множество **предикатов** над переменными 𝒱 (булевы выражения) |
| **𝒯 ⊆ 𝒮 × 𝒫 × ℝ⁺ × ℝ⁺ × 𝒮** | множество **переходов** вида ⟨s, p, t_min, t_max, s′⟩ |
| **𝒞** | множество **проверок** (sequence / timing / predicate / range) |
| **𝒢** | множество **гард-условий** на переменных, ассоциированных с переходами |
| **𝒥** | множество **кодов нарушений** (SEQ_MISS, TIME_UNDER, …) |

Это расширенная версия классического тайм-автомата Алура–Дилла (1994) с добавлением гард-условий и каталога нарушений; близко по выразительности к runtime-фрагменту STL Малера–Никовича (2004), но с существенной разницей: вместо непрерывных сигналов работаем с дискретными событиями исполнения сценария (event-log).

---

## 3. Структура наблюдаемой трассы

Трасса исполнения сценария — конечная последовательность пар

> **𝛔 = ⟨e₁, e₂, …, e_n⟩, где e_i = ⟨τ_i, σ_i, π_i, ν_i⟩**

| Поле | Что означает |
|---|---|
| **τ_i ∈ ℝ⁺** | временная метка события (секунды от начала прогона) |
| **σ_i ∈ Σ** | имя события из каталога (RUN_START, HOLD_END, MEAS_TICK, …) |
| **π_i: 𝒱 → значения** | снимок параметров процесса в момент события |
| **ν_i: 𝒱 → значения** | снимок измерительных сигналов в момент события |

Очевидно, τ₁ = 0 (RUN_START), τ_{i+1} ≥ τ_i, σ_n = RUN_END.

Трасса 𝛔 порождается логгером ИИС в формате JSONL (см. `event_log_format.md`).

---

## 4. Семантика приёма трассы

Трасса 𝛔 **принимается** моделью 𝓜 (обозначение **𝓜 ⊨ 𝛔**), если выполняется конъюнкция трёх условий:

### 4.1 Структурное условие (sequence)

Существует биекция между событиями трассы и шагами по автомату:

> ∀ i ∈ [1, n]: σ_i — событие, согласованное с состоянием σ̄(s_i) текущего шага в 𝓜, и переходы s_i → s_{i+1} ∈ 𝒯.

Кратко: каждый переход состояний в трассе разрешён моделью; ни один обязательный шаг не пропущен; порядок переходов соответствует FSM.

Нарушения этого условия → коды **SEQ_MISS** (пропуск обязательного шага), **SEQ_ORDER** (нарушенный порядок).

### 4.2 Временно́е условие (timing)

Для каждого пребывания в состоянии s длительностью d = τ_exit − τ_enter и для каждого исходящего перехода ⟨s, p, t_min, t_max, s′⟩, выбранного в трассе:

> t_min ≤ d ≤ t_max.

Нарушения → **TIME_UNDER** (d < t_min) или **TIME_OVER** (d > t_max).

### 4.3 Предикатное условие (predicate)

В момент каждого перехода ⟨s, p, t_min, t_max, s′⟩, выбранного в трассе, выполняется гард-условие:

> p(π_i ∪ ν_i) = true,

где π_i ∪ ν_i — объединённый снимок параметров и сигналов в момент τ_i.

Нарушения → **PRED_FAIL** (гард не выполнен), **RANGE_MISM** (значение вне допустимого диапазона), **N_TOO_LOW** (накопленное число отсчётов меньше N_min).

## 5. Вердикт

FLV-модуль формирует вердикт 𝓥(𝛔) по правилу:

```
если ∃ нарушение из 𝒥 с severity = "critical":
    𝓥(𝛔) = ⟨FAIL, violations⟩
иначе если ∃ нарушение из 𝒥 с severity = "warning":
    𝓥(𝛔) = ⟨OK_WITH_WARNINGS, violations⟩
иначе:
    𝓥(𝛔) = ⟨OK, ∅⟩
```

Где violations — структурированный список нарушений с привязкой к коду из 𝒥, конкретным численным фактам (expected vs actual) и ссылке на пункт нормативки.

---

## 6. Алгоритм проверки

```
Вход:    𝓜 (DSL-спецификация), 𝛔 (event-log JSONL)
Выход:   𝓥(𝛔) — вердикт + violations

1. Парсинг 𝛔: построение списка событий с нормализованным временем.
2. Реконструкция фактических переходов состояний: σ̄ = ⟨s_1, …, s_k⟩ + длительности d_j.
3. Sequence Matcher:
       3.1. Проверка обязательных шагов из 𝒞[kind=sequence].must_include.
       3.2. Проверка порядка переходов по 𝒯.
       3.3. Сбор нарушений SEQ_*.
4. Timing Matcher:
       Для каждого ⟨s, d⟩: проверка t_min ≤ d ≤ t_max.
       Сбор нарушений TIME_*.
5. Predicate Matcher:
       Для каждого перехода в 𝛔: вычисление гарда p(π ∪ ν).
       Дополнительно — проверка 𝒞[kind=predicate] и 𝒞[kind=range].
       Сбор нарушений PRED_*, RANGE_*, N_*.
6. Aggregator: формирует 𝓥(𝛔) согласно правилу из §5.
7. Reporter: машинно-читаемый JSON + Markdown-отчёт.
```

Сложность: 𝒪(n) по числу событий n трассы. Каждый matcher делает один линейный проход. Бенчмарк-цель: < 100 мс на n = 10 000.

---

## 7. Корректность модели

### Полнота 𝒞

Множество проверок 𝒞 покрывает все три условия приёма (§4): sequence, timing, predicate. Дополнительные виды (range, N) — частные случаи predicate.

### Непротиворечивость 𝒯

В DSL запрещены переходы из неопределённых состояний (validate.py проверяет from/to ∈ 𝒮). Это исключает «висячие» переходы и тупиковые состояния.

### Детерминизм 𝓥

При фиксированных 𝓜 и 𝛔 вердикт 𝓥(𝛔) — детерминированная функция. Это гарантирует юридическую воспроизводимость (см. ADR-002, Роль 2).

## 8. Пример прогона на бумаге

**Спецификация 𝓜 (S1, упрощённо):**

- 𝒮 = {INIT, HEAT, HOLD, MEASURE, POST}, s₀ = INIT.
- 𝒱 = {T, dT_dt, t_hold, N, T_set}.
- 𝒫: {device_ready, T ≥ T_min, |dT_dt| ≤ δ, N ≥ N_min}.
- 𝒯:
  - ⟨INIT, device_ready, 0, ∞, HEAT⟩
  - ⟨HEAT, T ≥ 50, 0, ∞, HOLD⟩
  - ⟨HOLD, |dT_dt| ≤ 0.02, 300, 600, MEASURE⟩
  - ⟨MEASURE, N ≥ 20, 0, ∞, POST⟩
- 𝒥: SEQ_MISS, TIME_UNDER, TIME_OVER, PRED_FAIL, N_TOO_LOW, RANGE_MISM.

**Трасса 𝛔 (фрагмент, упрощено до нужных переходов):**

| i | τ (с) | σ | s | π ∪ ν |
|---|---|---|---|---|
| 1 | 0 | RUN_START | INIT | {T_set=150} |
| 2 | 0.5 | INIT_END | INIT | {device_ready=true} |
| 3 | 0.5 | HEAT_START | HEAT | {} |
| 4 | 154.7 | HEAT_END | HEAT | {T=50.1} |
| 5 | 154.7 | HOLD_START | HOLD | {} |
| 6 | 184.7 | HOLD_END | HOLD | {t_hold=30, dT_dt=0.045} |
| 7 | 184.7 | MEAS_START | MEASURE | {} |
| ... | ... | ... | ... | ... |

**Прогон по алгоритму §6:**

1. Парсинг: 𝛔 = последовательность 7+ событий.
2. Реконструкция: переходы INIT→HEAT (длит. 154.2 с), HEAT→HOLD (длит. 30 с), HOLD→MEASURE.
3. Sequence Matcher: все обязательные шаги пройдены, порядок верен → нет нарушений.
4. Timing Matcher: для перехода HOLD→MEASURE длительность d = 184.7 − 154.7 = 30 с. По 𝒯: t_min = 300, t_max = 600. Условие 300 ≤ 30 нарушено → **TIME_UNDER** (expected ≥ 300, actual = 30).
5. Predicate Matcher: гард |dT_dt| ≤ 0.02 в момент HOLD_END. Факт: dT_dt = 0.045. Условие 0.045 ≤ 0.02 нарушено → **PRED_FAIL** (expected ≤ 0.02, actual = 0.045).
6. Aggregator: оба нарушения critical → 𝓥(𝛔) = ⟨FAIL, [TIME_UNDER, PRED_FAIL]⟩.

## 9. Связь с существующими формализмами

| Существующий формализм | Что общего | Что у нас иначе |
|---|---|---|
| Timed Automata (Алур–Дилл, 1994) | состояния, переходы с временными ограничениями, гард-условия | мы добавили каталог нарушений 𝒥 и явные условия приёма (§4.1-4.3); работаем в runtime, не в model checking |
| Signal Temporal Logic (Малер–Никович, 2004) | количественная семантика временных свойств | мы работаем с дискретными событиями, не с непрерывными сигналами |
| Metric Temporal Logic (Кoymans, 1990) | временные интервальные операторы | у нас встроенные t_min, t_max в каждом переходе вместо общих операторов until-within |
| Linear Temporal Logic | темпоральная корректность последовательностей | у нас расширение временными ограничениями + предикатами |

Уникальное отличие модели 𝓜 — **прикладной DSL**, заточенный под измерительные методики ГОСТ-семантики; явный каталог 𝒥 нарушений с привязкой к нормативке. Это и есть позиция, занимаемая FLV в обзорной таблице (см. `01_Источники/05_Аналоги/_сравнение.md`).

## 10. Открытые расширения

- **Континуальная семантика для STL.** В версии 2.0 — добавление непрерывного канала сигналов рядом с дискретными событиями; STL-предикаты как опциональные дополнения к булевым гардам.
- **Стохастические гард-условия.** Сейчас все предикаты — детерминированные булевы. Расширение с вероятностными порогами — отдельный научный трек.
- **Multi-stand параллелизм.** Сейчас модель — для одиночного прогона; расширение для синхронизированных стендов — версия 2.0.
- **Иерархические состояния (statecharts).** Для более сложных методик с вложенными режимами.
