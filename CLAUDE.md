### 2.5 Использование субагентов (Agent tool)

- **Когда обязательно делегировать в субагент:**
  - Прогон литературы (≥ 3 PDF/статей) с конспектом — основной поток будет забит цитатами, контекст быстро переполнится.
  - Большой WebFetch (страница сайта, документация инструмента) → суммаризация в 200-300 слов.
  - Парсинг больших ToolSearch/MCP-выдач (> 30 KB) — отдать subagent'у, вернуть короткий список.
  - Параллельные исследования (например: «найти 5 русских статей по метрологии» + «найти 5 английских по STL» — два subagent'а в одном вызове).
  - Любая «копай глубже»-задача, где неочевидно сколько шагов потребуется (поиск по структуре кафедры, поиск платных статей в открытом доступе).
- **Когда не нужно:** одиночные WebSearch/Read/Edit; уточняющие вопросы пользователю (это прямо `AskUserQuestion`).
- **Типы агентов:** `general-purpose` (по умолчанию), `Explore` (узконаправленный поиск кода/файлов), `Plan` (архитектура).
- **Брифинг агенту обязательно содержит:**
  1. Что именно найти/сделать (с конкретным выходом).
  2. Почему это нужно (контекст ВКР: тема, фаза, какая глава).
  3. Что уже исключено или известно (чтобы агент не переделывал работу).
  4. Лимит ответа («ответ под 300 слов», «список из ≤ 10 пунктов»).
- **Если в чате видно, что задача потенциально требует углубления (даже если это неочевидно), но углубление съест много токенов — сразу делегировать в subagent.** Лучше потратить один агент-вызов, чем забить главный контекст промежуточными выдачами.

### 2.5 Стандарты оформления

- **ГОСТ 7.32-2017** — структура отчёта о НИР: титульный лист, реферат (5-15 ключевых слов, объём в страницах), содержание, введение, основная часть, заключение, список источников, приложения.
- **ГОСТ 7.0.5-2008** — библиографическая ссылка.
- **ГОСТ 2.105-95** — общие требования к текстовым документам.
- **Шрифт:** Times New Roman 14, межстрочный 1.5, поля: левое 30 мм, правое 10, верх/низ 20.
- **Объём бакалаврской ВКР:** 60-80 страниц (без приложений), при тебе ~70.

### 2.6 Конвенция git-коммитов

**Цель.** Каждый коммит — атомарный инженерный шаг. По `git log` мы должны иметь возможность восстановить логику работы и собрать черновик пояснительной записки автоматически (для каждой главы — выборка соответствующих коммитов).

#### 2.6.1 Формат сообщения

```
<phase>(<scope>): <субъект коротко в imperative> [<artifact-ref>]

<тело: что сделано, какое инженерное решение принято, почему>

Refs: <стандарт/источник, если опираемся на конкретный>
Artifact: <относительный путь к созданному/изменённому файлу>
Phase-progress: <фаза статус>  ← опционально, если меняли статус в ВКР_План.md
```

#### 2.6.2 Префиксы фаз (`<phase>`)

| Prefix | Фаза |
|---|---|
| `p0` | подготовка инфраструктуры (план, CLAUDE.md, скелет репо) |
| `p1` | сбор источников |
| `p2` | спецификация и архитектура |
| `p3` | симулятор стенда |
| `p4` | модуль FLV |
| `p5` | эксперименты и статистика |
| `p6` | пояснительная записка |
| `p7` | презентация и защита |
| `meta` | мета-изменения (CLAUDE.md, README, план, .gitignore, корректировки структуры) |
| `fix` | багфиксы и опечатки (без привязки к фазе) |
| `wip` | незавершённое (использовать редко; завершённый шаг — без `wip`) |

#### 2.6.3 Скоупы (`<scope>`)

Внутри каждой фазы — короткий тег модуля/раздела:

| Фаза | Возможные скоупы |
|---|---|
| p1 | `gost`, `fgos`, `mgtu`, `lit`, `analog`, `biblio` |
| p2 | `dsl`, `schema`, `fsm`, `event-log`, `arch`, `viol`, `validate` |
| p3 | `thermal`, `sensor`, `scenario`, `injector`, `logger`, `viz`, `cli`, `scpi` |
| p4 | `loader`, `parser`, `seq-matcher`, `time-matcher`, `pred-matcher`, `verdict`, `reporter`, `cli`, `online`, `tests`, `bench` |
| p5 | `runner`, `analyze`, `stats`, `figures`, `report`, `notebook` |
| p6 | `intro`, `ch1-обзор`, `ch2-метод`, `ch3-реал`, `ch4-эксп`, `concl`, `biblio`, `format`, `figures` |
| p7 | `slides`, `doklad`, `qa`, `demo` |
| meta | `plan`, `claude`, `readme`, `repo` |

#### 2.6.5 Правила атомарности

- **Один коммит = один логический шаг.** Не смешивай добавление источника с написанием главы.
- **Не коммить «всё разом».** Если за сессию сделано 5 шагов — 5 коммитов.
- **Перед коммитом** — обновляй `ВКР_План.md` (если статус фазы поменялся), журнал изменений.
- **Тело коммита обязательно** для всего, что не тривиально (например, для `meta(readme)` можно одной строкой).

#### 2.6.6 Сборка дайджеста для пояснительной записки

Из истории `git log` мы можем за минуту получить:
- список всех источников: `git log --grep "^Refs:" --pretty=%B | grep "^Refs:"`;
- список всех артефактов фазы N: `git log --grep "^pN" --pretty="%h %s"`;
- хронологию работы: `git log --pretty="%ad %s" --date=short`.

Эти выборки → черновик «методической части» главы 3 ПЗ (как делалась работа).

#### 2.6.7 Когда работаю я (Claude)

- Каждый завершённый артефакт — оформляю в виде комплекта изменений и оставляю Даниле инструкцию вида: `git add <files> && git commit -m "..."`.
- Если git-репо позволяет коммит из бот-сессии (без virtiofs-блокеров) — коммит сам.
- В сообщениях `Phase-progress` указываю формально, чтобы можно было `git log --grep "Phase-progress" --pretty=%B` восстановить ход выполнения.

## 3. Workflow одной сессии

1. **Открой `ВКР_План.md`** → найди ближайшую `pending`/`in_progress` фазу.
2. **Открой соответствующий Phase-файл** (`Phase1`/`Phase2`).
3. **Выполни задачу**: код в подпапке, текст в Markdown (если глава ПЗ — в `06_ПЗ/`).
4. **Обнови статус** в `ВКР_План.md` (статус, дата, артефакт).
5. **Допиши строку в журнал изменений**.
6. **Если открылся новый блокер** — задай Даниле вопрос через `AskUserQuestion` или зафиксируй в `99_Артефакты/blockers.md`.

## 4. Что НЕ делать

- Не помещать актуальный план в `/outputs/` или временные папки.
- Не редактировать read-only исходники (`Концепт.md`, `план ВКР.md` и т.п.) — они зафиксированы как референс.
- Не писать главы ПЗ напрямую в `.docx` без сначала Markdown-черновика в `06_ПЗ/`.
- Не использовать LaTeX как финальный формат (договорились — .docx).
- Не выдумывать источники. Только реальные, с верифицируемой ссылкой.
- Не молча менять scope. Любое сокращение/расширение фазы фиксируется в `ВКР_План.md` с обоснованием.
