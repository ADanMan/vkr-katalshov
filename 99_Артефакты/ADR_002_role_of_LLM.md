# ADR-002 — Роль LLM в архитектуре FLV

> Дата: 2026-05-03. Статус: principal direction. Подлежит уточнению на старте Phase 4 (реализация модуля FLV).

## Контекст

В исходном плане ВКР (`99_Артефакты/00_Исходные/план_ВКР.md`, раздел 17 «Риски и меры») LLM фигурировал так: *«LLM используется только для первичного извлечения требований в DSL-скелет; подтверждение и верификация — детерминированные»*. Это безопасное стартовое позиционирование, но оно недоиспользует уникальный потенциал современных LLM в задачах, связанных с обработкой неструктурированных нормативных текстов и формированием человекочитаемых объяснений.

Уточнённое позиционирование (после обсуждения с Данилой 2026-05-03): **система валидации через LLM** — это не «LLM делает всё», а **гибридная архитектура**, где LLM применяется на двух хорошо отграниченных шагах с детерминированной перепроверкой. Это и есть **новизна метода относительно RTAMT/Breach/UPPAAL**, у которых LLM-слой отсутствует, и относительно «GPT-codereview», у которого нет формального подкрепления.

## Решение — три роли LLM

### Роль 1. Препроцессор «текст методики → DSL-скелет»

**Вход:** текстовый фрагмент ГОСТ/МИ/ТУ (раздел или подраздел методики измерения).
**Выход:** YAML-черновик DSL-спецификации (`states`, `transitions`, `parameters`, `checks`).

**Процедура:**
1. LLM получает: текст методики + DSL-промпт + JSON Schema + примеры (few-shot).
2. LLM возвращает черновик YAML.
3. Черновик автоматически валидируется по `flv_dsl.schema.json` (см. `02_Спецификация/validate.py`).
4. Если валидация не прошла — LLM получает сообщение об ошибке + один retry.
5. Если прошла — DSL переходит к **этапу human-in-the-loop**: инженер просматривает, утверждает или правит.
6. **Только после явного утверждения инженером** DSL принимается как нормативная модель.

**Защита от галлюцинаций:**
- JSON Schema отсекает синтаксически неверные структуры.
- Семантическая проверка `validate.py` (from/to в states, related_check в check ids).
- HITL-подпись инженера фиксирует ответственность за корректность модели.

**Зачем LLM здесь:**
- Извлечение структуры из неструктурированного текста — задача, на которой LLM **сильнее любого rule-based парсера**, особенно для разнообразных нормативных формулировок (ГОСТ, МИ, ТУ, инструкции вендоров).
- Альтернатива (ручное составление DSL для каждой методики инженером) — медленная и дорогая.

### Роль 2. Детерминированный matcher (БЕЗ LLM)

**Вход:** утверждённый DSL + JSONL-трасса исполнения сценария.
**Выход:** структурированный verdict (`OK` / `FAIL`) + список violations с фактами (expected vs actual).

LLM здесь **категорически не применяется.** Все проверки — детерминированные алгоритмы (sequence, timing, predicate matchers).

**Почему так:**
- Вердикт должен быть воспроизводим: один и тот же DSL + одна и та же трасса = один и тот же результат, без вариативности от температуры sampling, runs или версии модели.
- Метрологическая прослеживаемость требует юридически документируемой проверки. LLM не даёт юридической гарантии.
- Если научрук или комиссия спросит «почему вы доверяете методу?» — ответ должен быть в виде формального доказательства, а не «LLM решил».

### Роль 3. Постпроцессор «verdict → human-readable объяснение»

**Вход:** структурированный verdict + контекст (DSL + соответствующий фрагмент трассы).
**Выход:** Markdown-фрагмент с объяснением для инженера/метролога/ревизора:
- Какой пункт нормативки нарушен (с цитатой из исходного текста методики).
- Что именно произошло в трассе (с числами и временами).
- Какова рекомендация (повторить, скорректировать сценарий, эскалировать).

**Процедура:**
1. LLM получает: verdict + violation codes + extract из исходного DSL и трассы.
2. LLM формирует объяснение в строго заданном Markdown-формате.
3. Объяснение проходит финальную регулярку-проверку на наличие ссылок на конкретные числа из трассы (анти-галлюцинационная страховка).

**Защита от галлюцинаций:**
- LLM не может изобрести нарушение — он работает с уже сформированным детерминированным verdict.
- Все числа должны быть **процитированы** из контекста; пост-фильтр отбрасывает фразы без цифр-якорей.
- В формальном FLV-сертификате ссылки на verdict идут с machine-readable JSON; LLM-объяснение — только справочное приложение.

### Опционально — Роль 4 (после защиты)

**LLM-as-judge для пограничных случаев семантической эквивалентности.** Когда инженер написал сценарий с другой структурой, чем DSL, но семантически тот же (например, переименовал состояния, объединил переходы) — LLM может предложить, эквивалентны ли они. Это исследовательский трек, не входит в бакалаврскую ВКР.

```
                  │ Текст методики        │
                  │ (ГОСТ/МИ/ТУ)          │

                             │
                  ┌──────────▼────────────┐
                  │ LLM РОЛЬ 1            │

                  │ + JSON Schema валид.  │

                  └──────────┬────────────┘
                             │
                  ┌──────────▼────────────┐

                  │ инженер утверждает    │

                             │
                  ┌──────────▼────────────┐
                  │ DSL (утверждённый)    │

                             │

```

## Сравнение со смежными подходами

| Подход | LLM | Формальная проверка | Объяснимость | Гарантии |
|---|---|---|---|---|
| RTAMT, Breach (STL-мониторы) | нет | да | низкая | формальная, привязана к STL |
| UPPAAL (model checking) | нет | да (design-time) | низкая | формальная, нет runtime |
| GPT-codereview, Copilot | да | нет | высокая | нет |
| Requirements Gateway (NI) | нет | нет (только трассировка) | средняя | нет |
| **FLV (наш метод)** | **да (роли 1, 3)** | **да (роль 2)** | **высокая** | **формальная (детерм. matcher) + вспомогательная (LLM-объяснение)** |

**Уникальное преимущество:** LLM — там, где он силён (текст → структура, структура → текст), и нет — там, где нужна юридическая воспроизводимость (формальное решение).

## Ограничения и риски

1. **LLM может пропустить или исказить детали** в Роли 1. **Меры:** Schema-валидация + HITL-подпись инженера + явный лог промптов и версий модели.
2. **Стоимость и латентность LLM** в production-ИИС. **Меры:** LLM срабатывает только на этапе подготовки методики (Роль 1, один раз) и пост-обработки вердикта (Роль 3, один раз на прогон). На горячем пути (исполнение сценария + matcher) LLM нет.
3. **Зависимость от LLM-провайдера.** Архитектурное решение — единая абстракция через **OpenRouter** (https://openrouter.ai), что даёт прозрачное переключение между моделями без переписывания кода. Для эксперимента Phase 5 фиксируем 5 моделей разных классов и поставщиков: **Gemini Flash 3** (Google), **GPT 5.4** (OpenAI), **Claude Sonnet 4.6** (Anthropic), **DeepSeek V4 Flash** (DeepSeek), **Qwen3.6 Flash** (Alibaba). Это позволяет провести сравнение качества LLM-ролей (precision/recall извлечения DSL) на 5 точках и подкрепить выбор данными, а не маркетингом. **Меры:** в каждой LLM-вызывающей функции — `model_id` как параметр; версии моделей фиксируются на дату начала Phase 5 и логируются в каждый прогон.
4. **Галлюцинации в Роли 3.** **Меры:** анти-галл фильтр на уровне regex / structured-output; vetting корпуса промптов; каждый ключевой числовой факт привязан к JSON-vердикту.
5. **Юридический статус LLM-вывода в РФ.** **Меры:** в ПЗ явно указано, что вердикт основан на детерминированном matcher'е; LLM-объяснение — справочное.
