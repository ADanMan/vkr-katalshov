# ADR-003 — Инженерный стек симулятора и интерактивная визуализация

> Дата: 2026-05-03. Статус: principal direction. Применяется в Phase 3.

## Контекст

Изначальный план Phase 3 (`02_Спецификация/simulator_arch.md`, версия после Phase 2) предполагал минималистичный Python-симулятор: явный Эйлер для теплопередачи, ручная гауссова добавка для шума, matplotlib для статики, Three.js для 3D. Этого достаточно как proof-of-concept, но недостаточно как **инженерно-доказательная работа**, которую защищает выпускник К2 «Информационно-измерительная техника и технологии». Метрологическая часть — обработка единиц, propagation погрешностей, регуляризация контура — должна делаться **корректно**, а не «на коленке».

Кроме того, защита и ПЗ выигрывают от **интерактивной визуализации**: и для 3D-демонстрации стенда, и для дашборда, где видны живые сигналы T(t), состояние FSM и вердикт FLV.

## Решение

Phase 3 строится на профессиональных научно-инженерных библиотеках и двух интерактивных вью.

### Ядро симуляции

| Слой | Библиотека | Версия | Замена ручному коду |
|---|---|---|---|
| ODE-интегрирование | `scipy.integrate.solve_ivp` | scipy ≥ 1.11 | вместо ручного Эйлера — RK45 (по умолчанию), LSODA для жёстких систем, контроль локальной ошибки `rtol=1e-6` |
| Дискретные события + время | `simpy` | ≥ 4.1 | вместо ручного `while`-цикла — event-driven SIMulation в PYthon; естественный язык для FSM-сценариев с задержками |
| Единицы измерения | `pint` | ≥ 0.23 | строгая работа с °C, K, °C/s, Ом, мВ, Гц; ловит ошибки типа «складываем секунды и градусы» на этапе разработки |
| Метрологическая неопределённость | `uncertainties` | ≥ 3.1 | автоматическая propagation погрешностей через нелинейную характеристику PT100 + АЦП — прямая иллюстрация ГОСТ 8.207-76 |
| Контур управления | `python-control` | ≥ 0.9 | PID-регулятор для поддержания T_set; передаточные функции, Bode/Nyquist для секции «теоретическая часть» ПЗ |
| Сигналы | `scipy.signal` | ≥ 1.11 | окна, фильтры, спектры; шумовая модель датчика как окрашенный шум, а не чистый AWGN |

### Интерактивная визуализация

Две независимые view, каждая со своей задачей:

| View | Стек | Цель | Артефакт для ПЗ/защиты |
|---|---|---|---|
| **3D-стенд** | `pyvista` (геометрия) + `trame` (web UI) + `vtk` | живая 3D-сцена термокамеры с цилиндром-образцом, цвет которого меняется от температуры; в браузере, с поворотом, зумом, скрабером времени | скриншот в ПЗ + HTML-демо в презентацию |
| **FLV Dashboard** | `plotly.dash` | live-чарт T(t), FSM-state в реальном времени, индикатор verdict OK/FAIL, таблица violations | embedded в защиту как «вживую» демо |

Оба компонента — самостоятельные сервисы; могут запускаться по отдельности либо вместе.

### Статические figures для ПЗ

Не отказываемся от статики — она нужна в Word:

- `matplotlib` + `seaborn` (стиль `paper`, шрифты TNR-совместимые) — все 7 графиков Phase 5.
- `pyvista.export_html()` для 3D-снимков с возможностью встраивания в slides; `screenshot()` для PNG в Word.

## Альтернативы и почему отвергнуты

| Альтернатива | Почему отвергнута |
|---|---|
| Чистый numpy + Euler + matplotlib | Поверхностно для бакалавра-приборостроителя; математически корректные методы есть «из коробки», их использование = плюс на защите |
| Modelica / OpenModelica | Профессиональный инструмент моделирования, но overkill и порог входа неоправдан для одной термокамеры |
| Simulink | Закрытый, проприетарный, требует MATLAB-лицензии; не воспроизводимо для open-source демонстрации |
| Streamlit | Хороший для UI, но заточен под data-apps, не под сцену с 3D-объектами; pyvista интегрируется хуже |
| Three.js напрямую | Делает 3D-визуализацию, но нет связности с Python-моделью; pyvista + trame даёт ту же интерактивность из Python без писания JS |

## Последствия

### Позитивные

1. Глава 3 ПЗ («Программная реализация») получает раздел про инженерный стек — иллюстрирует владение современным Python-научным стеком.
2. Метрологическая корректность работы с единицами и неопределённостями — закрывает компетенции **ОПК-2** и **ПК-2** ФГОС 12.03.01.
3. Контур управления PID — даёт раздел «теоретическая часть» с передаточной функцией, Bode-диаграммой, оценкой устойчивости.
4. Интерактивные view доступны и в офлайн, и в браузере — комиссия может «потыкать» во время защиты.

### Негативные

1. Объём зависимостей пакета вырос с 5 до ~15 — нужен `pyproject.toml` с фиксированными версиями.
2. Время на освоение `simpy` и `python-control` — ~1-2 дня. Заложено в бюджет Phase 3.
3. `pyvista` требует VTK; локальный запуск без проблем, но в headless-окружении (CI) — нужен `xvfb` или эквивалент. На Mac работает из коробки.

### Риски

| Риск | Меры |
|---|---|
| Освоение библиотек выше времени | Жёсткий timebox: если на конец дня 2 не получается базовый thermal_model + sensor с pint+uncertainties — возвращаемся к минимальному стеку |
| pyvista/trame в Cowork-сессии | Эта сессия не запускает GUI; артефакты создаются как код + демо-скрипты, реальный запуск — на машине автора |
| Отсутствие GPU для 3D | pyvista работает software-rendering; нагрузка терпимая для одного цилиндра |

## План реализации (детализация Phase 3)

```
3.1 ADR-003 + обновление simulator_arch.md         ← этот шаг
3.2 pyproject.toml + структура пакета + sim/config.py
3.3 thermal_model.py (scipy.integrate.solve_ivp + pint)
3.4 uncertainty_model.py (uncertainties для цепи PT100→АЦП)
3.5 sensor.py (PT100 по ГОСТ 6651 + АЦП + uncertainty)
3.6 control_loop.py (python-control: PID, передаточная функция, Bode)
3.7 scenario_runner.py (simpy-движок FSM)
3.8 injector.py (7 типов нарушений)
3.9 event_logger.py (JSONL по схеме Phase 2)
3.10 scenarios/*.yaml (8 сценариев)
3.11 cli.py (CLI: одиночный прогон, batch, seed)
3.12 viz/3d_pyvista.py (3D-сцена + trame)
3.13 viz/dashboard_dash.py (Plotly Dash)
3.14 tests/ (pytest для каждого модуля)
3.15 README + примеры запуска + скриншоты
```

15 атомарных коммитов; пустой каркас → работающий симулятор → интерактивные view.
