# Научное письмо — паттерны под главы ВКР

ВКР технического направления = жанр научно-инженерного отчёта. Близок к IMRAD (Introduction, Methods, Results, Discussion), но с инженерной спецификой: добавляются «Аналитический обзор», «Программная реализация», более объёмное обсуждение методики.

## Принцип «бабочки»

Каждая глава — «бабочка»:
- **Шеечка**: вступительный абзац (что в главе будет, зачем).
- **Тело**: подразделы с фактами, расчётами, схемами.
- **Заключительная шеечка**: краткие выводы по главе (3-7 предложений), мостик к следующей главе.

Это позволяет читать и работу целиком, и каждую главу в отрыве — комиссия часто читает выборочно.

## Введение — паттерн

```
1. Актуальность (1 стр.):
   - Контекст: ИИС — основа автоматизации измерений.
   - Проблема: семантический разрыв между методикой и кодом.
   - Цена ошибки: время, реагенты, доверие к результатам.
   - Тренды: рост сложности экспериментов, ужесточение метрологических требований.
2. Цель работы (1-3 предложения).
3. Задачи (5 шт., из плана работы) — нумерованный список.
4. Объект исследования (1 предложение).
5. Предмет исследования (1 предложение).
6. Научная новизна (3-4 пункта, нумерованных).
7. Практическая значимость (4-5 пунктов).
8. Методы исследования (1 абзац).
9. Структура работы (1 абзац: «В первой главе... Во второй главе... В заключении...»).
```

Объём — 4-6 страниц.

## Глава 1 «Аналитический обзор»

Цель — обосновать, что наша задача (а) актуальна, (б) не решена существующими методами.

```
1.1 Что такое ИИС, какие есть классы, какова метрологическая надёжность.
    [3-4 источника — учебники]
1.2 Программные модели измерительных протоколов: история, языки (LabVIEW, TestStand, Python, SCPI).
    [3-5 источников — статьи + документация NI]
1.3 Существующие подходы к верификации:
    1.3.1 Статический анализ (TestStand SA, VI Analyzer)
    1.3.2 Трассируемость (Requirements Gateway)
    1.3.3 Runtime Verification и temporal logic (Maler, Bartocci, RTAMT, Breach)
    [8-12 источников]
1.4 Семантический разрыв — что это и почему критично.
    [Свой текст с примером wait(30) vs wait(300)]
1.5 Постановка задачи (формальная) — мостик к главе 2.
```

Объём — 15-20 страниц. Структура «расширяющейся воронки»: от широкого контекста к конкретной нише.

## Глава 2 «Метод»

Цель — формально описать FLV-метод так, чтобы по тексту его можно было воспроизвести.

```
2.1 Концепция: вход = нормативная модель, выход = вердикт. Иллюстрация — общая схема FLV-петли.
2.2 DSL: грамматика, типы сущностей, пример полной спецификации для S1.
2.3 Timed FSM: математическая нотация, семантика, теорема о соответствии трассы.
2.4 Алгоритм сопоставления: pseudocode для sequence/timing/predicate matchers.
2.5 Каталог нарушений: 7 кодов, формальное определение каждого.
2.6 Интеграция с ИИС: контур online/offline, диаграмма архитектуры.
```

Объём — 15-20 страниц. Должна быть «самодостаточной» — теоретическая база работы.

## Глава 3 «Реализация»

Цель — показать, что метод реализуем и реализация работает.

```
3.1 Архитектура модуля FLV (UML-схема компонентов).
3.2 Симулятор стенда (физическая модель + 3D + инжектор).
3.3 Детали реализации matchers (со ссылками на key listings, полные — в приложениях).
3.4 3D-визуализация и UI (скриншоты).
3.5 Тестирование (pytest coverage, бенчмарки).
```

Объём — 10-15 страниц. Не перегружать листингами в основной части — только ключевые фрагменты, остальное в приложения.

## Глава 4 «Эксперимент»

Цель — доказать эффективность метода.

```
4.1 Методика эксперимента (дизайн, переменные, контроль).
4.2 Описание стендов S1 (термокамера) и S2 (SCPI).
4.3 Каталог инъекций нарушений.
4.4 Результаты — таблицы и графики.
4.5 Статистическая обработка (McNemar, Wilcoxon, Cohen's d, CI).
4.6 Обсуждение — почему результаты получились такие, ограничения, threats to validity.
```

Объём — 15-20 страниц. Это «ядро» защиты — сюда обычно идут вопросы комиссии.

## Заключение

```
1. Сводка решённых задач (mapping на 5 задач из введения).
2. Главные численные результаты (1-2 цифры).
3. Подтверждение научной новизны (повтор, но усиленно).
4. Практическая значимость на конкретном примере.
5. Перспективы развития.
```

Объём — 2-3 страницы. Не нужно вводить новые факты — только выводы по уже сказанному.

## Стиль

- **Безличные конструкции** (а не «я сделал», «мы получили»): «было реализовано», «получены результаты», «проведено сравнение».
- **Прошедшее время** для описания выполненной работы: «модуль FLV был спроектирован», «прогон состоял из...».
- **Настоящее время** для теории и метода: «Timed FSM состоит из...».
- **Будущее время** только в перспективах в заключении: «в дальнейшем планируется...».
- **Без жаргона** — каждое сокращение определено при первом упоминании. ИИС — Информационно-измерительная система; STL — Signal Temporal Logic; и т.д.
- **Цифры** — в основном тексте словами до 9 («пять стендов»), цифрами от 10. В таблицах и формулах — цифрами всегда.
- **Единицы измерения** — через неразрывный пробел: `300 с`, `150 °C`, `100 МГц`.

## Антишаблоны (что НЕ писать)

- ✗ «На сегодняшний день» — клише, удалять.
- ✗ «Активно развивается» — пустая фраза, требует конкретики.
- ✗ «Бурно развивается» — то же.
- ✗ «Я считаю», «Мы считаем» — для технической работы недопустимо.
- ✗ Длинные предложения > 30 слов — разбивать.
- ✗ Эмоционально окрашенные слова: «революционный», «прорывной», «уникальный» — использовать строго по факту, иначе удалять.
- ✗ Чрезмерное использование двоеточия и тире.

## Объём и темп

- 1 страница ≈ 1700-2200 знаков с пробелами при TNR 14, 1.5, поля 30/10/20/20.
- Глава 15-20 стр. ≈ 30-40 тыс. знаков.
- За день можно написать 5-7 стр. качественного черновика, если фактура собрана. Если факты гуглятся параллельно — 2-3 стр.

## Использование скилла `humanizer`

После написания каждой главы — прогон через скилл `humanizer` (Anthropic), чтобы убрать AI-следы:
- Em-dash overuse.
- Однотипные «-ing» конструкции (не для русского, но переводы).
- «Rule of three» (триплеты «X, Y и Z» в каждом абзаце).
- AI-vocabulary («delve», «navigate», «underscore» в переводах).
