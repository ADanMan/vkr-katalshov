# Симулятор измерительного стенда S1 — физика и параметры

Стенд S1 — термокамера со стабилизацией температуры и измерением через PT100. Это «канонический» пример из метрологии для отработки методики измерений с условием стабилизации.

## Физическая модель

### Уравнение теплопереноса (1-й порядок)

```
dT/dt = (T_set - T) / τ + ξ(t)
```

Где:
- `T(t)` — температура объекта (образца), °C.
- `T_set` — заданная температура камеры, °C (обычно 150).
- `τ` — постоянная времени системы, с (обычно 60).
- `ξ(t)` — гауссовский шум с σ ≈ 0.05 °C, среднее 0.

Это упрощённая модель «один объект, одна тепловая ёмкость, конвективный теплообмен». Достаточно для демонстрации FLV; в реальности есть градиенты, тепловые мосты, паразитные ёмкости — добавим в обсуждении главы 4 как threats to validity.

### Дискретизация

Шаг моделирования `dt = 0.1` с (10 Гц). Отсчёты температуры — каждые 1 с (1 Гц), что соответствует типичной частоте логгера в LabVIEW.

Метод интегрирования: явный Эйлер (для простоты) или RK4 (если потребуется лучшая точность).

## Сенсор PT100

### Зависимость R(T)

ГОСТ 6651-2009, IEC 60751:

```
R(T) = R_0 * (1 + a*T + b*T²)              для T ≥ 0 °C
R(T) = R_0 * (1 + a*T + b*T² + c*(T-100)*T³)  для T < 0 °C
```

Где:
- R_0 = 100 Ом (сопротивление при 0 °C).
- a = 3.9083e-3.
- b = -5.775e-7.
- c = -4.183e-12 (для отрицательных T).

В нашем диапазоне 50-200 °C используем только верхнюю формулу.

### Обратная функция (для пересчёта R → T)

```python
T = (-a + sqrt(a**2 - 4*b*(1 - R/R_0))) / (2*b)
```

## АЦП

- Разрядность: 16 бит.
- Диапазон: 0-500 °C (после нормирующего преобразования R → V → ADC code).
- Квантование: ~0.0076 °C (= 500/65536).
- Дрейф нуля: 1e-3 °C/час (учитываем, если прогон > 1 часа).
- Шум АЦП: σ = 0.5 LSB ≈ 0.004 °C.

Сумма шумов сенсора + АЦП — итоговый шум измерения ≈ 0.05 °C.

## Параметры сценария по умолчанию

| Параметр | Значение | Что значит |
|---|---|---|
| `T_set` | 150 °C | заданная температура |
| `τ` | 60 с | постоянная времени |
| `T_min` | 50 °C | порог HEAT → HOLD |
| `δ_stable` | 0.02 °C/с | порог стабилизации (для HOLD → MEASURE) |
| `t_hold_min` | 300 с | минимальная выдержка |
| `t_hold_max` | 600 с | максимальная выдержка |
| `N_min` | 20 | минимум отсчётов в MEASURE |
| `T_set_max` | 250 °C | предельная температура камеры |
| `f_sample` | 1 Гц | частота отсчётов |

## FSM сценария

```
INIT (device_ready==true) → HEAT (T >= T_min) → HOLD (|dT/dt| <= δ И t_hold >= 300) → MEASURE (N >= N_min) → POST → [*]
```

## Каталог инъекций нарушений

| Код | Имя сценария | Что меняется |
|---|---|---|
| `s1_correct.yaml` | эталон | всё по норме |
| `s1_seq_miss` | пропуск шага | пропускаем HOLD, идём INIT→HEAT→MEASURE |
| `s1_seq_order` | нарушение порядка | MEASURE раньше HOLD |
| `s1_time_under` | недовыдержка | t_hold = 30 c вместо ≥ 300 |
| `s1_time_over` | перевыдержка | t_hold = 700 c вместо ≤ 600 |
| `s1_pred_fail` | предикат не выполнен | переход HOLD→MEASURE при |dT/dt| = 0.05 (> δ) |
| `s1_n_too_low` | мало измерений | N = 5 вместо ≥ 20 |
| `s1_range_mism` | вне диапазона | T_set = 700 °C (> T_set_max) |

Каждый сценарий — YAML с полями `name`, `inject` (тип), `params_override` (что подменяется).

## Воспроизводимость

- Все случайные процессы (шум сенсора, АЦП) параметризованы `seed`.
- При фиксированном seed прогон полностью детерминирован.
- В `runs/_metadata.csv` сохраняем seed для каждого прогона.

## 3D-визуализация

Минимум: HTML с Three.js, прозрачная термокамера-куб, цветной образец-цилиндр (синий → красный по температуре), live-chart рядом, подсветка состояния FSM.

План Б (если Three.js «съест» время): matplotlib `FuncAnimation` + одна 3D-картинка через pyvista для иллюстрации в ПЗ.

## SCPI-симулятор (S2, опционально)

Если успеваем — второй стенд через `pyvisa-sim`:
- Источник напряжения (V_SRC) + мультиметр.
- SCPI-команды: `*RST`, `CONF:VOLT:DC`, `RANGE 10`, `READ?`.
- FSM: `SET_RANGE → SETTLING (t≥τ) → MEASURE×N → FILTER`.

## Реалистичность параметров

Параметры τ=60 c, σ=0.05 °C, t_hold=300 c — типичные для лабораторной термокамеры (например, типа DKB-30, Binder MK-56). В обсуждении главы 4 ссылаемся на конкретный коммерческий аналог для realism check.
