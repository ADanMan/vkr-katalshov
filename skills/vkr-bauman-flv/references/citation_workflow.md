# Workflow поиска и оформления источников

Цель — для каждой ссылки в ПЗ иметь верифицированный источник в `BIBLIO.bib` + правильно оформленный пункт списка по ГОСТ 7.0.5.

## Шаг 1. Поиск

Параллельно по 4 базам:

| База | Когда использовать |
|---|---|
| Google Scholar | первый заход; подбор по ключевым словам |
| Semantic Scholar | для англо-источников + подбор похожих по графу цитирований |
| arXiv | для preprints, особенно по STL/MTL/RV |
| eLibrary.ru | для российских журналов и диссертаций |
| ResearchGate | для скачивания закрытых статей (через автора) |

Запросы — см. `flv_domain_knowledge.md`, раздел «Поисковые запросы».

## Шаг 2. Извлечение метаданных

Для каждой найденной работы получаем DOI/arXiv ID/PMID. Затем — автоматическое извлечение через CrossRef API:

```bash
# Один источник
python scripts/doi_to_bibtex.py 10.1007/978-3-540-30206-3_12 >> BIBLIO.bib

# Список из текстового файла
cat dois.txt | xargs -I{} python scripts/doi_to_bibtex.py {} >> BIBLIO.bib
```

Скрипт делает запрос к `https://api.crossref.org/works/{DOI}` и возвращает BibTeX с полями: `author`, `title`, `journal`, `year`, `volume`, `pages`, `doi`. Если DOI не найден в CrossRef — пробует Semantic Scholar API; если и там нет — заполняет minimal-skeleton с пометкой `[VERIFY]`.

## Шаг 3. Валидация

```bash
python scripts/validate_bibtex_gost.py BIBLIO.bib
```

Скрипт проверяет:
- Все обязательные поля по типу (article: author, title, journal, year, volume, pages).
- Корректность формата года (4 цифры).
- Наличие DOI/URL для электронных источников.
- Уникальность ключей.
- Кириллические/латинские конфликты в авторах.

Output — список ошибок, чтобы исправить вручную.

## Шаг 4. Цитирование в Markdown

В черновиках глав используем pandoc-стиль:

```markdown
Метод STL впервые описан в работе [@maler2004monitoring]. Подход к online-мониторингу
развит в [@deshmukh2017robust; @nickovic2020rtamt].
```

При финальной сборке `.docx` через pandoc:

```bash
pandoc 02_Глава1.md \
  --bibliography=BIBLIO.bib \
  --csl=assets/gost-r-7-0-5-2008.csl \
  --reference-doc=assets/reference_gost.docx \
  -o 02_Глава1.docx
```

Pandoc сам заменит `[@key]` на `[N]` и сгенерирует список источников по ГОСТ.

## Шаг 5. Финальная сборка библиографии

После сборки всех глав — единый список в порядке упоминания:

```bash
python scripts/build_bibliography.py 06_ПЗ/draft/*.md \
  --bibtex BIBLIO.bib \
  --output 06_ПЗ/draft/Литература.md \
  --style gost-7.0.5
```

Скрипт сохраняет порядок первого упоминания и форматирует каждую запись по соответствующему ГОСТ 7.0.5 шаблону (см. `gost_7.0.5_citation.md`).

## Pro-tip: чтение PDF быстро

Для каждой статьи делаем 3-уровневый конспект:

1. **Один абзац** в `_README.md` папки — что работа делает, чем нам полезна, какой ключ BibTeX.
2. **Цитируемые места** — фразы, которые планируем переиспользовать (в кавычках, с указанием страницы).
3. **Метки релевантности** — какие главы ВКР опираются на эту работу, например, `tags: [гл1, обзор, STL]`.

Это позволяет потом за 5 минут выбрать нужное при написании главы.

## Закрытые статьи (платные Springer/IEEE)

Если нет доступа к ЭБС МГТУ:
1. ResearchGate — часто авторы выкладывают.
2. Sci-Hub — в учебных целях допустимо, но проверь институциональные правила.
3. Запрос автору по email (Mendeley/ORCID) — обычно отвечают.
4. Если совсем недоступно — отметить в `_README.md` как `[CLOSED]` и в Phase 1-отчёте записать как пробел.

## Норма по объёму библиографии

- ВКР бакалавра: 20-40 источников (Положение МГТУ).
- Наша цель: ≥ 25, оптимум 30-35.
- Распределение по типам: ~50% статьи, ~25% книги/учебники, ~15% стандарты, ~10% электронные ресурсы.
