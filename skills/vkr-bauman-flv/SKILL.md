## 3. Жёсткие принципы (не нарушать)

1. **Источник истины — `ВКР_План.md` в корне репо.** Любая работа начинается с его чтения. После завершения шага — обновляй статус фазы и журнал изменений.
2. **Никогда не выдумывай источники.** Все ссылки — реальные, верифицированные через DOI/CrossRef/Semantic Scholar. Если найти не можешь — пометь `[ИСТОЧНИК?]` и вернись.
3. **Главы ПЗ пишутся сначала в Markdown** в `06_ПЗ/draft/<NN_имя>.md`, и только финал конвертируется в `.docx`. Никогда не пиши сразу в Word.
4. **Финальный формат — .docx.** LaTeX используется только как справочный материал из `references/bachelor_diploma_structure.md`. Не предлагай LaTeX как итоговый.
5. **Кодовые артефакты (Python/YAML/JSON Schema) — только ASCII в именах файлов** (для git-стабильности). Текст ПЗ — на русском, имена .md можно с кириллицей.
6. **Делегируй в субагенты тяжёлую работу:** прогон ≥ 3 PDF, парсинг крупного MCP-вывода, параллельные литературные поиски. Подробнее — в CLAUDE.md, §2.5.
7. **Каждый завершённый артефакт оформляется git-коммитом.** Конвенция — в CLAUDE.md, §2.6 (`pN(scope): subject` + `Refs:` + `Artifact:` + `Phase-progress:`). Атомарно, по одному логическому шагу.

## 4. Workflow одной сессии

```
1. Прочти CLAUDE.md и ВКР_План.md.
2. Найди ближайшую `pending`/`in_progress` фазу.
3. Открой соответствующий ВКР_План_PhaseN_*.md.
4. Сделай шаг → положи артефакт в нужную подпапку.
5. Обнови статус в ВКР_План.md (фаза, дата, артефакт, журнал).
6. Сформулируй сообщение коммита по конвенции из CLAUDE.md §2.6.
7. Если открылся блокер — записать в `99_Артефакты/blockers.md` или задать `AskUserQuestion`.
```

## 6. Стандарты оформления (минимум, который надо держать в голове)

- **Шрифт:** Times New Roman 14 pt, межстрочный 1.5.
- **Поля:** левое 30 мм, правое 10 мм, верхнее/нижнее 20 мм.
- **Абзац:** отступ 1.25 см, выравнивание по ширине.
- **Нумерация страниц:** арабская, в правом верхнем углу. Счёт от титульного листа, нумерация — с введения.
- **Заголовки разделов:** прописными, центр, без точки.
- **Иллюстрации:** подпись внизу — «Рисунок N — Название».
- **Таблицы:** подпись сверху — «Таблица N — Название».
- **Ссылки:** в квадратных скобках `[N]`, нумерация — в порядке упоминания.
- **Список литературы:** по ГОСТ 7.0.5-2008.

Полный чек-лист — в `references/gost_7.32_structure.md`.

## 7. Структура ПЗ (целевая, под ГОСТ 7.32 + Положение МГТУ)

1. Титульный лист (по шаблону К4).
2. Реферат (5-15 ключевых слов, объём, иллюстрации, источники, приложения — на 1 странице).
3. Содержание (автогенерация в Word).
4. Перечень сокращений и условных обозначений.
5. Введение (4-6 стр.) — актуальность, цель, задачи, объект, предмет, новизна, практическая значимость, методы, структура.
6. Глава 1 «Аналитический обзор» (15-20 стр.) — ИИС и метрологическая надёжность, программные модели методик, существующие подходы (TestStand SA, VI Analyzer, RTAMT, Breach), формальные методы (STL/MTL/Timed Automata), семантический разрыв, постановка задачи.
7. Глава 2 «Метод и формальная модель» (15-20 стр.) — концепция FLV, DSL, Timed FSM, алгоритм сопоставления, каталог нарушений, интеграция с ИИС.
8. Глава 3 «Программная реализация» (10-15 стр.) — архитектура FLV-модуля, симулятор стенда, детали matchers, 3D-визуализация, тестирование.
9. Глава 4 «Экспериментальное исследование» (15-20 стр.) — методика, стенды S1 и S2, инъекции, результаты, статистика, обсуждение.
10. Заключение (2-3 стр.).
11. Список использованных источников (≥ 25).
12. Приложения (DSL, JSON Schema, листинги, доп. графики).

**Целевой объём:** 65-75 стр. основного текста + приложения.

## 12. Скрипты (исполняются по необходимости)

- `scripts/doi_to_bibtex.py` — DOI/PMID/arXiv ID → запись BibTeX через CrossRef API. Использование: `python doi_to_bibtex.py 10.1007/978-3-540-30206-3_12 >> BIBLIO.bib`.
- `scripts/validate_bibtex_gost.py` — проверка BibTeX-записей на соответствие минимальным полям ГОСТ 7.0.5-2008 (автор, год, название, выходные данные, страницы для статей).
- `scripts/md_to_docx_gost.py` — конвертация Markdown глав в `.docx` со стилем ГОСТ 7.32-2017 (TNR 14, 1.5, поля 30/10/20/20). Использует pandoc + reference.docx.
- `scripts/mermaid_to_png.py` — рендер `.mmd` в `.png` 300 DPI для вставки в ПЗ. Требует mermaid-cli.
